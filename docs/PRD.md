# PRD: Dual Commerce Native + Uniapp Architecture Sample

## 背景

本项目面向 Android 架构师或高级 Android 工程师面试，展示一个 toB + toC 电商系统在 Android Native 与 Uniapp 混合架构下的边界设计。项目不是完整业务系统，不追求真实交易闭环，而是追求架构可解释、模块边界清晰、工程治理完整。

## 目标

- 展示 Android Native 如何承载核心交易链路、系统能力、性能治理和隐私合规。
- 展示 Uniapp 如何承载高频变化、运营配置和轻量商家工作台页面。
- 展示 Native-Uniapp Bridge 如何统一能力调用、权限边界和错误协议。
- 展示 mock API 与 Model Contract 如何保证双端字段、状态和错误语义一致。
- 展示高并发、弱网、性能、隐私、AI 场景和 CI/CD 在架构层的设计能力。
- 支持面试时用 1 分钟、5 分钟和追问方式讲清楚架构取舍。

## 非目标

- 不实现完整电商 App。
- 不接真实后端。
- 不实现真实登录和真实支付。
- 不接真实第三方 SDK。
- 不实现复杂推荐、搜索、风控、库存或营销引擎。
- 不一次性生成完整 Android 与 Uniapp 代码。
- 不把核心交易链路放入 Uniapp。

## 用户与场景

- toC 用户：浏览商品、查看详情、加入购物车、提交订单、查看售后入口。
- toB 商家：查看工作台、管理商品轻量信息、筛选订单、查看库存状态、生成营销文案。
- 运营人员：配置活动页、营销页、优惠楼层和活动模板。
- 面试官：关注候选人是否能讲清楚边界、取舍、风险和扩展路径。
- 开发团队：通过文档、ADR 和小 issue 保持协作一致性。

## 核心需求

- Android Native Shell：提供启动、路由、mock 登录态、核心交易链路、SDK 门面、性能监控和隐私合规。
- Uniapp Business Layer：提供活动页、营销页、商家工作台、商品管理轻量页、订单筛选页、AI 文案页和配置化页面。
- Bridge：统一 Native 能力暴露，定义请求、响应、错误、权限和版本兼容协议。
- API SDK：封装 mock API，约束网络错误、业务错误、降级和缓存策略。
- Model Contract：定义商品、购物车、订单、支付占位、售后、库存、营销和 AI 场景的共享模型语义。
- Performance：定义启动、商品列表、图片、网络、Uniapp 页面和 Bridge 指标。
- Privacy：定义权限、SDK 清单、敏感 API 门面、埋点脱敏和授权状态管理。
- AI Scenarios：定义智能客服、商品问答、标题生成、营销文案、售后话术和异常分析的 mock 场景。

## 验收标准

- 架构文档能解释 Native、Uniapp、Shared Core、Backend API、Monitor / Track / Privacy / CI/CD 的关系。
- 每个关键边界都有 ADR 说明背景、决策、取舍和后果。
- 文档明确哪些能力不得实现为真实登录、真实支付、真实 SDK 或真实后端。
- 文档能支撑面试中的 1 分钟概述、5 分钟展开和追问回答。
- 后续开发任务可以从文档拆成小 issue，不需要一次性实现完整项目。

## 风险与约束

- 混合架构最容易出现调用散落，因此 Bridge 必须作为唯一边界。
- 双端模型容易漂移，因此 API 与 Model Contract 必须先于页面实现。
- 运营页面容易侵入交易链路，因此核心交易动作必须由 Native 收口。
- 隐私合规容易被 SDK 或业务绕过，因此敏感能力必须统一门面。
- AI 输出存在幻觉和合规风险，因此只能作为辅助建议，不自动影响交易。

## 面试价值

这个 PRD 的重点不是“做多少功能”，而是“为什么这样做”。它让候选人可以围绕业务稳定性、动态交付、Bridge 治理、模型契约、弱网性能、隐私合规、AI 风险和团队协作讲出完整架构判断。
