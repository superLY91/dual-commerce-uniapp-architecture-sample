# Project Context

## 项目背景

本项目是一个面向“电商 toB + toC Android 原生 + Uniapp 双端架构师岗位”的面试架构样板项目。

它不是完整电商 App，也不以跑通真实交易为目标。项目的核心价值是用有限代码和文档展示架构判断：哪些能力放 Android Native，哪些页面放 Uniapp，Native 与 Uniapp 如何通过 Bridge 协作，以及性能、隐私合规、CI/CD、AI 应用和团队协作如何进入架构设计。

## 面试岗位要求

目标岗位需要展示以下能力：

- Android Native 架构设计能力，包括模块拆分、分层架构、数据边界和可测试性。
- Uniapp 页面组织能力，包括页面分层、mock API、状态管理和跨端复用边界。
- Native 与 Uniapp 混合架构设计能力，包括 Bridge 协议、调用链路、错误处理和权限边界。
- 电商业务理解能力，能区分核心交易链路和高频变化运营页面。
- 性能治理意识，包括启动性能、页面加载、资源治理、WebView/Uniapp 容器性能和监控埋点。
- 隐私合规意识，包括权限申请、隐私弹窗、SDK 合规、数据最小化和审计记录。
- CI/CD 与工程协作意识，包括质量门禁、构建流水线、代码规范、文档规范和 issue 拆分。
- AI 工程应用意识，包括如何用 AI 辅助需求拆分、代码审查、测试补全和文档维护。

## 领域术语

- toC：面向消费者的购物端，关注商品浏览、购物车、下单、订单查看和售后入口。
- toB：面向商家、运营或企业客户的管理端，关注商品管理、订单处理、库存协同、营销配置和数据看板。
- Android Native：Android 原生部分，负责核心能力、核心链路、设备能力、性能敏感场景和合规边界。
- Uniapp：跨端页面层，适合承载高频变化、运营配置驱动、对性能和安全要求相对较低的页面。
- Bridge：Native 与 Uniapp 之间的统一通信层，负责协议定义、能力暴露、参数校验、错误返回和权限控制。
- Core：Android 通用能力模块，例如网络 mock、日志、基础 UI、线程调度、配置和工具能力。
- Feature：Android 业务页面模块，每个 feature 只关注自己的页面和用例。
- Privacy：隐私合规模块，统一管理权限申请、隐私状态、合规拦截和审计口径。
- Repository：数据访问抽象层，用于隔离 mock 数据源、未来真实数据源和业务用例。
- Mock API：用于面试样板演示的数据接口，不连接真实后端。
- ADR：Architecture Decision Record，用于记录关键架构决策、背景、取舍和后果。

## Android Native 和 Uniapp 的职责边界

Android Native 负责：

- App 壳工程与模块化骨架。
- 核心交易链路的架构表达，例如商品详情关键动作、购物车、下单入口、订单核心状态。
- 性能敏感能力，例如启动、路由、容器管理、缓存和关键页面加载。
- 设备与系统能力，例如权限、存储、相机、定位、通知等能力的统一封装。
- 隐私合规边界，包括权限申请、隐私确认、SDK 初始化时机和合规审计。
- Bridge 能力注册、调用分发、参数校验和安全控制。

Uniapp 负责：

- 高频变化页面，例如活动页、营销页、店铺装修页、运营配置型页面。
- 页面组织、mock API、轻量状态管理和跨端展示逻辑。
- 通过 Bridge 调用 Native 暴露的受控能力。
- 展示混合架构中动态页面如何和 Native 壳协作。

边界原则：

- 核心交易链路不放 Uniapp。
- 需要强合规、强性能、强稳定性的能力优先放 Native。
- 高频变化、展示驱动、运营配置型页面优先放 Uniapp。
- Uniapp 不直接访问 Native 内部模块，只通过统一 Bridge 交互。
- 所有接口先使用 mock，不接真实后端。

## 电商核心业务模块

本项目用于表达以下电商模块的架构边界，不要求完整实现：

- 用户与身份：只做游客态或 mock 用户态，不做真实登录。
- 商品：商品列表、商品详情、商品状态和价格展示的 mock 结构。
- 搜索与分类：用于展示页面组织和数据流，不接真实搜索服务。
- 购物车：用于表达核心交易链路边界，可用 mock 数据表示。
- 订单：用于表达订单状态流转、详情查看和售后入口。
- 支付：只保留架构占位和风险说明，不做真实支付。
- 营销：适合放入 Uniapp 的活动、优惠、运营配置型页面。
- 店铺与商家：toB 场景下的商品管理、订单处理、库存协同和数据看板。
- 消息与通知：只做架构说明或 mock，不接真实推送 SDK。
- 隐私与合规：权限、隐私弹窗、SDK 初始化控制和合规审计。

## 不做什么

- 不做完整电商 App。
- 不做真实登录。
- 不做真实支付。
- 不接真实后端。
- 不接真实第三方 SDK。
- 不实现复杂推荐、搜索、风控或营销引擎。
- 不引入复杂依赖或重型框架。
- 不一次性生成完整项目。
- 不把核心交易链路下沉到 Uniapp。
- 不用业务堆砌掩盖架构边界。

## 面试表达目标

面试表达时应围绕以下问题组织：

- 为什么这是 Native + Uniapp 的双端混合架构，而不是纯 Native 或纯 H5。
- 哪些模块属于稳定核心能力，哪些模块属于高频变化页面。
- Bridge 如何避免调用分散、权限失控和协议不可维护。
- Privacy 模块如何控制权限、SDK 初始化和合规风险。
- Repository 和 mock API 如何让样板项目不依赖真实后端仍能表达架构。
- CI/CD 如何保障架构样板的质量，包括 lint、测试、构建和文档检查。
- AI 如何参与需求拆分、issue 生成、测试补齐、文档维护和代码审查。
- 团队协作中如何通过小 issue、ADR 和上下文文档保持一致性。
