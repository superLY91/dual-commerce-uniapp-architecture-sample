# ADR 0004: 性能与弱网策略

## 状态

Accepted

## 背景

电商 App 对启动、商品列表、图片加载、网络请求、购物车和订单提交有明显性能要求。混合架构还额外引入 Uniapp 页面加载和 Bridge 调用耗时。如果不提前定义指标和策略，性能问题会分散在各业务页面中。

## 决策

性能治理围绕启动、列表、图片、网络、Uniapp 页面和 Bridge 六类场景展开：

- 启动优化：延迟非必要初始化，隐私确认前不初始化受限 SDK，首屏只加载必要配置。
- 商品列表优化：分页、占位、复用、轻量模型和主线程保护。
- 图片优化：尺寸裁剪、缓存、占位、弱网低清图和滑动节流。
- 网络优化：统一超时、请求去重、读请求有限重试、写请求依赖幂等 token。
- Uniapp 页面性能：分块加载活动配置，减少首屏组件数量，集中封装 Bridge 调用。
- Bridge 监控：记录调用耗时、成功率、错误码和 traceId。

## 高并发与弱网策略

- 防重复提交：关键按钮进入提交中状态，结果返回前不重复触发。
- 幂等 token：订单提交、支付占位和售后申请使用 token 表达服务端防重。
- 请求去重：同参数读请求短时间内合并。
- 超时重试：读请求可以有限重试，写请求不盲目自动重试。
- 降级开关：活动楼层、推荐、AI 和非核心营销组件可降级。
- 本地缓存：商品基础信息、活动配置和工作台配置可缓存。
- 状态兜底：异常时给出明确状态和恢复路径。

## 指标

- App 启动耗时。
- 首屏加载耗时。
- 商品列表 FPS。
- 图片加载成功率和耗时。
- API 耗时、错误率和重试率。
- Bridge 调用成功率、错误码分布和耗时。
- Uniapp 页面加载耗时。
- 降级开关命中率。

## 备选方案

- 只在出现问题后优化：成本低，但面试中无法体现治理体系。
- 直接引入复杂 APM：能力强，但违反样板项目不接真实 SDK 的边界。

## 取舍理由

本项目用文档和 mock 指标表达性能治理思路，不接真实 APM SDK。这样既能展示架构能力，也符合轻量样板的约束。

## 后续影响

- 后续代码可以预留 monitor 接口，但不能接真实第三方 SDK。
- 核心链路提交类动作必须考虑防重和幂等。
- Uniapp 页面上线前要考虑首屏、Bridge 次数和降级方案。

## 面试讲法

我不会只说“做启动优化”，而是把启动、列表、图片、网络、Uniapp 和 Bridge 都变成可观测指标，再结合弱网和高并发策略做兜底。
