# ADR 0005: 隐私合规边界

## 状态

Accepted

## 背景

电商 App 会涉及设备能力、位置、相机、相册、通知、剪贴板、埋点和第三方 SDK。即使本项目不接真实 SDK，也需要展示合规意识：权限不能散落申请，敏感 API 不能被业务绕过，SDK 初始化不能早于用户授权。

## 决策

建立 `privacy` 作为统一合规边界：

- 权限统一申请：业务页面只声明业务意图，由 `privacy` 模块判断是否申请系统权限。
- SDK 清单：记录 SDK 名称、用途、采集字段、初始化时机和关闭策略。
- 敏感 API 门面：定位、设备标识、相机、相册、剪贴板、通知等能力通过统一门面访问。
- 埋点脱敏：手机号、地址、订单号、用户标识等敏感字段不得明文上报。
- 授权状态管理：隐私协议确认、权限授权、拒绝、撤回和降级状态统一维护。
- Bridge 权限校验：Uniapp 调用敏感能力时必须经过 Bridge 和 privacy 校验。

## 备选方案

- 每个业务模块自行申请权限：开发快，但合规风险高。
- 只依赖系统权限弹窗：无法覆盖隐私协议、SDK 初始化和审计要求。
- 完全不设计 SDK 边界：样板简单，但无法体现真实 App 治理能力。

## 取舍理由

隐私合规必须是架构边界，不是页面实现细节。集中到 `privacy` 模块后，业务可以保持简单，审计和面试表达也更清晰。

## 后续影响

- 任何敏感能力都不能直接从业务模块调用系统 API。
- 隐私同意前不能初始化受限 SDK。
- 埋点方案必须先定义脱敏规则。
- Uniapp 页面不能绕过 Bridge 访问敏感能力。

## 面试讲法

我把隐私合规做成模块边界：权限、SDK、敏感 API、埋点和授权状态都收口。这样面试官能看到我不是事后补弹窗，而是在架构层防止违规调用。
