# ADR 0001: Native 与 Uniapp 职责边界

## 状态

Accepted

## 背景

电商 App 同时存在稳定核心链路和高频变化页面。商品详情关键动作、购物车、订单提交和支付占位对稳定性、安全性、性能和异常兜底要求高；活动页、营销页、商家轻量工作台和 AI 文案页变化频繁，要求快速交付。

如果全部 Native 化，运营迭代成本高；如果全部 Uniapp 化，核心交易链路的性能、合规和稳定性风险变高。

## 决策

- Android Native 负责启动、mock 登录态、支付占位、推送占位、商品核心流、购物车、订单提交、Native SDK 门面、性能监控和隐私合规。
- Uniapp 负责活动页、营销页、商家工作台、商品管理轻量页、订单筛选页、AI 文案生成页和配置化页面。
- 核心交易链路不放 Uniapp。
- Uniapp 调用 Native 能力必须经过统一 Bridge。

## 备选方案

- 纯 Native：稳定性强，但运营和 toB 轻量页面迭代慢。
- 纯 Uniapp：交付快，但核心交易链路边界弱，性能和合规风险更高。
- 多套散落 WebView：短期灵活，但长期协议、权限和监控不可控。

## 取舍理由

该拆分保留 Native 对交易链路和系统能力的控制，同时让 Uniapp 承担变化快、展示强、配置化的页面。面试中能清楚表达“稳定性与迭代速度”的平衡。

## 后续影响

- 后续 Android 模块要体现 `core`、`feature`、`bridge`、`privacy`、`domain`、`data` 的边界。
- 后续 Uniapp 目录要体现 `pages`、`api`、`store`、`bridge`、`utils` 的分层。
- 新页面归属必须先判断是否属于核心交易链路。

## 面试讲法

我用 Native 扛稳定核心，用 Uniapp 扛高频变化。这样不是技术混搭，而是按业务风险拆分：交易链路追求稳定、安全、性能和合规，运营和商家轻量页面追求快速迭代。
