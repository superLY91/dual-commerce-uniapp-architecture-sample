# ADR 0003: API SDK 与 Model Contract

## 状态

Accepted

## 背景

Native 与 Uniapp 同时表达商品、购物车、订单、库存、营销和 AI 场景。如果两个端各自定义字段和状态，后续很容易出现含义漂移，例如订单状态、库存状态、活动状态和错误码在双端解释不同。

本项目不接真实后端，但仍需要模拟真实团队中的接口契约治理。

## 决策

- 所有 API 使用 mock 数据，不连接真实后端。
- Native 与 Uniapp 面向同一套 API 语义和模型契约。
- Model Contract 先定义领域含义，再映射到页面模型。
- Repository 隔离数据源，未来可从 mock 替换为真实实现，但业务用例不直接依赖数据源细节。
- API 错误分为网络错误、业务错误、权限错误、降级错误和未知错误。

## 核心模型

- Product：商品基础信息、价格、库存状态、上下架状态、营销标签。
- Search：关键词、筛选项、排序、分页和空状态。
- Cart：购物车项、勾选状态、数量、价格快照和失效状态。
- Order：订单状态、提交结果、支付占位状态、售后入口和异常状态。
- PaymentPlaceholder：支付方式占位、支付结果占位和风险说明。
- Inventory：库存数量、预警状态、锁库存占位和更新来源。
- Marketing：活动类型、优惠规则、楼层配置和降级状态。
- AIContent：输入上下文、生成候选、人工确认状态和风险提示。

## 备选方案

- 页面直接消费 mock JSON：速度快，但契约不可维护。
- Native 与 Uniapp 分别建模型：灵活，但字段含义容易漂移。
- 过早引入复杂 OpenAPI 工具链：规范强，但不符合样板项目轻量原则。

## 取舍理由

轻量 Model Contract 能展示架构意识，又不会让样板项目被工具链复杂度吞没。它保留未来演进空间，同时满足面试解释。

## 后续影响

- 新增业务字段时先说明领域含义和状态枚举。
- 双端页面不得直接创造不在契约中的核心状态。
- 真实后端接入不是当前目标，但 Repository 边界要能支持替换。

## 面试讲法

即使是 mock 项目，我也会先定义模型契约，因为混合架构最大的问题不是接口能不能调通，而是双端对同一个业务状态是否理解一致。
